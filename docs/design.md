# 设计文档：知识发现与推理引擎

> 从模糊的原始文档到可执行的知识产物。
>
> 本文档提炼自 AGENTS.md 中的设计方案，补充了每层流程的异常路径与失败模式。
> 代码实现状态见 README.md，两者对照阅读。

## 1. 分层架构

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌─────────────┐
│   Data      │ ──▶ │ Information  │ ──▶ │  Knowledge  │ ──▶ │   Wisdom    │
│ (原始文档)   │    │ (三元组+置信度)│    │ (图谱+规则)  │    │ (代码/问答)  │
└─────────────┘    └──────────────┘    └─────────────┘    └─────────────┘
       │                   │                   │                  │
       ▼                   ▼                   ▼                  ▼
   Markdown/日志        JSONL 中间态        图数据库 + 规则      Go structs
   YAML frontmatter    智能标记结果         Datalog 推理         FAQ/API 设计
```

### 设计原则

- **知识发现优先**：不强求初始本体，允许数据保留模糊性，在使用中逐渐浮现结构。
- **不确定性显式化**：每条候选事实携带置信度，不丢弃低置信度数据，留待累积或人工确认。
- **声明式推理**：Datalog 保证终止且顺序无关，避免命令式规则的不确定性。
- **知识即代码**：最终输出可落地的代码结构、API 设计、问答对等智慧产物。

## 2. 数据流详解

### 2.1 数据层（Data）—— 原始原料

**输入格式**：Markdown 文件（可含 YAML frontmatter），JSONL 日志，结构化事件。

**阳光路径**：
1. 读取文件
2. 解析 YAML frontmatter 与正文
3. 提取概念列表、关系列表、规则陈述
4. 输出结构化的文档对象

**异常路径**：

| 异常 | 处理策略 |
|------|----------|
| YAML frontmatter 格式错误 | 忽略 frontmatter，继续按纯 Markdown 处理，记录警告 |
| 文件编码非 UTF-8 | 尝试 GBK/Shift-JIS 回退编码；均失败则跳过，记入处理失败列表 |
| 空文件 | 跳过，记录警告 |
| 超大文件（>10MB） | 分块读取，逐块交付给下游；或提示用户手动分割 |
| 二进制文件（图片/PDF） | 拒绝处理，返回明确错误类型及建议的预处理工具 |

### 2.2 信息层（Information）—— 智能标记

**核心操作**：将非结构化文本转化为带置信度的语义三元组。

**阳光路径**：
1. 接收文档对象
2. 分块（chunking），每块独立评估
3. 调用 LLM 提取候选三元组（subject, predicate, object）
4. 为每条三元组计算置信度并绑定来源上下文
5. 输出 `triples.jsonl`

```
{ "id": "extract-001", "source": "docs/order.md",
  "confidence": 0.65, "subject": "订单", "predicate": "包含",
  "object": "商品", "context": { "sentence": "订单包含多个商品" },
  "verification": "unverified" }
```

**异常路径**：

| 异常 | 处理策略 |
|------|----------|
| LLM 返回非 JSON 格式 | 重试 1 次；仍失败则跳过，记录完整请求/响应用于人工排查 |
| LLM 超时 | 指数退避重试（100ms → 500ms → 2s → 放弃） |
| LLM 返回 JSON 但缺少必填字段 | 用空值补全缺失字段，标记 `verification: "malformed"` |
| 分块边界截断语义 | 分块策略采用重叠窗口（overlap 10%），尾部语义不完整块标记 `truncated: true` |
| 置信度普遍偏低（<0.3） | 不丢弃，写入低置信度暂存区，累积确认后触发重新评估 |
| 同一来源反复失败 | 累计 3 次失败后从队列中移除该来源，通知人工审查 |

### 2.3 知识层（Knowledge）—— 结构化入库与推理

**核心操作**：高置信度三元组写入图数据库，Datalog 规则推理演绎出新知识。

**阳光路径**：
1. 过滤：保留 `confidence >= threshold` 的三元组
2. 写入：批量写入图数据库（CozoDB / Neo4j）
3. 推理：加载图谱事实 + Datalog 规则，执行查询
4. 演绎：输出推导出的新事实

**异常路径**：

| 异常 | 处理策略 |
|------|----------|
| 数据库连接失败 | 重试 3 次（1s → 5s → 30s），全部失败则回退到本地 JSON 文件存储 |
| 写入冲突（相同三元组已存在） | 用 max(现有置信度, 新置信度) 合并，记录合并历史 |
| 规则循环依赖 | Datalog 保证终止，但需在规则加载时做静态循环检测，提前报错 |
| 规则推导空结果 | 不视为错误，返回空集并记录 query 与时间戳 |
| 推理超时 | 为每条规则设置超时上限（默认 10s），超时视为不成立 |
| 规则文件中存在未定义谓词 | 编译时检测并报错，列出未定义谓词列表 |

**阈值策略**：

```
规则：
  - 高置信度 >= 0.7 → 自动入库
  - 中置信度 0.4~0.7 → 写入待确认区，等待人工或额外证据累积
  - 低置信度 < 0.4 → 保留在 JSONL 中，不自动入库

阈值可配置、可按来源/谓词独立设置。
```

### 2.4 智慧层（Wisdom）—— 应用输出

**核心操作**：基于推理结果生成可执行的代码结构、API 设计、问答对。

**阳光路径**：
1. 查询推理结果
2. 根据目标类型（code / api / faq）选择生成器
3. 渲染输出

**异常路径**：

| 异常 | 处理策略 |
|------|----------|
| 推理结果不足（<3 条事实） | 返回"数据不足，需要更多信息"，不生成空壳代码 |
| 生成器模板不存在 | 返回可用生成器列表，报错说明 |
| 输出路径无写入权限 | 降级到 stdout 打印，提示用户 |
| 生成的代码语法无效 | 尝试用 `go fmt` 等格式化工具修复；失败则附注"需人工审查" |

## 3. 关键设计决策

| 问题 | 决策 | 理由 |
|------|------|------|
| 推理语言 | Datalog（Mangle 或 CozoDB 内建 Datalog） | 保证终止、顺序无关、声明式 |
| 图数据库 | 待定（CozoDB / Neo4j 二选一） | CozoDB 自带 Datalog 可简化技术栈，Neo4j 生态更成熟。选型前需原型验证性能 |
| 中间交换格式 | JSONL | 行级追加、流式处理、每行独立可解析 |
| 不确定性模型 | 置信度 + 来源上下文 + 审核状态 | 保留溯源路径，支持增量累积 |
| 分块策略 | 重叠窗口（10% overlap），尾部标记 | 减少边界截断导致的语义丢失 |

## 4. 状态与追踪

每条候选事实的生命周期状态机：

```
unverified ──▶ verified (自动或人工)
    │
    ├──▶ malformed (格式异常)
    ├──▶ pending_merge (与已有事实冲突)
    └──▶ discarded (低于阈值且无累积价值)
```

## 5. 未解决的问题

- 图数据库选型（CozoDB vs Neo4j）——需原型验证写入/查询性能后决定
- Mangle 与 CozoDB 内建 Datalog 的功能重叠——如果选 CozoDB，Mangle 不再需要
- 分块大小与 LLM 上下文窗口的适配策略——需根据实际模型调整
- 多源冲突消解的优先级规则——目前用 max 置信度合并，更复杂的策略待定
